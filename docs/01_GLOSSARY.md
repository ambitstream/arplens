# Glossary

**Project:** ArpLens

**Version:** 2.3 (Frozen)

---

# Purpose

This document defines the terminology used throughout the ArpLens documentation.

Every technical term should have exactly one meaning.

All documents should reference these definitions consistently.

---

# Arpeggio

A repeating ordered sequence of pitched notes generated by applying an arpeggiator style to a set of input notes.

For the MVP, only deterministic monophonic arpeggiator styles are supported.

---

# Standard Arpeggio

An arpeggio that can be fully explained by one supported Style Registry entry.

Unsupported or manually played note sequences are **not** considered standard arpeggios.

---

# Input Notes

The notes held by the musician that are fed into the arpeggiator.

Example:

```
C
D#
G
```

Input Notes are not the same as the generated sequence.

---

# Generated Sequence

The ordered sequence of notes produced by applying:

- Input Notes
- Style
- Octaves

through the Style Registry.

This is the sequence reproduced by the Preview Engine.

---

# Style

The traversal rule that determines how the Input Notes are played.

Examples:

- Up
- Down
- UpDown
- DownUp

Styles are defined exclusively in the Style Registry.

---

# Style Registry

The single source of truth for all supported arpeggiator styles.

The registry generates note sequences.

It is shared by:

- Analysis Engine
- Preview Engine
- Manual Editor
- Unit Tests

---

# Step

The smallest timing unit of an arpeggio.

Each Step contains one or more note indices.

In the MVP, every Step contains exactly one note.

Future versions may support multiple notes per Step.

---

# Cycle

One complete repetition of an arpeggio pattern.

Every generated sequence consists of exactly one Cycle.

The Analysis Engine attempts to detect one Cycle from the selected audio.

---

# Focus Region

A user-selected one-minute section extracted from long audio files.

If the uploaded audio exceeds one minute, the user first selects a Focus Region.

All further analysis is performed only inside this region.

---

# Loop

The final 3–20 second fragment selected by the user for analysis.

The Loop should contain one seamlessly repeating arpeggio phrase.

Analysis quality depends directly on loop quality.

---

# Step Grid

The internal timing grid estimated by the Analysis Engine.

Every detected note is assigned to the nearest Step.

The Step Grid is derived from the audio itself.

It is not imported from MIDI.

---

# Quantization

The process of aligning detected note events to the estimated Step Grid.

The output is a deterministic note sequence suitable for style matching.

---

# Pitch Event

A note detected by Spotify Basic Pitch.

Each Pitch Event includes:

- pitch
- onset
- offset
- confidence

Pitch Events are intermediate analysis data.

They are never shown directly to the user.

---

# Hypothesis

A candidate explanation of the observed arpeggio.

Each Hypothesis consists of:

- Input Notes
- Style
- Octaves

The Analysis Engine evaluates multiple hypotheses before selecting the best one.

---

# Joint Hypothesis Enumeration

The process of generating candidate arpeggiator configurations.

Input Notes, Style and Octaves are solved together rather than independently.

---

# Style Matching

The process of comparing the detected Cycle against sequences generated by the Style Registry.

Matching is:

- deterministic
- rotation invariant
- edit-distance based

---

# Rotation

A cyclic shift of a generated sequence.

Example:

Original:

```
C E G
```

Rotation:

```
E G C
```

The Analysis Engine compares all valid rotations.

---

# Quantized Sequence

The ordered note sequence produced after Quantization.

It represents the Analysis Engine's interpretation of the performed arpeggio.

---

# Step Duration

The duration of one Step in seconds, derived from the Step Grid.

Step Duration is available whenever a Quantized Sequence exists.

It is part of the Result DTO so that partial results remain playable.

---

# Arpeggio Sandbox

An alternate entry point that lets the user experiment with
arpeggiator settings without uploading audio.

The Sandbox seeds the editable model with default values instead
of deriving it from a Result DTO.

It is an in-page mode, not a separate page or route.

---

# Detected Sequence

The final sequence presented to the user.

It may be:

- generated from the winning Style (complete result)
- reconstructed from the quantized sequence (partial result)

---

# Partial Result

A valid analysis result where some parameters cannot be determined reliably.

Partial Results are preferred over incorrect Complete Results.

---

# Complete Result

A result where the Analysis Engine successfully determines:

- Input Notes
- Style
- Rate
- Octaves
- BPM

with sufficient confidence.

---

# Confidence

The Analysis Engine's assessment of the reliability of the reconstructed result.

Displayed values:

- High
- Medium
- Low

Confidence reflects the weakest reliable stage of the analysis pipeline.

---

# Preview Engine

The subsystem responsible for synthesizing and playing the reconstructed arpeggio.

It performs no analysis.

It simply reproduces the sequence provided by the Analysis Engine or Style Registry.

---

# Analysis Engine

The deterministic computational core of ArpLens.

It reconstructs arpeggiator settings from audio.

The Analysis Engine contains no UI code.

---

# Result DTO

The immutable object returned by the Analysis Engine.

It contains all analysis results required by the UI and Preview Engine.

User edits never modify the original Result DTO.

---

# Engine Version

The version identifier of the Analysis Engine implementation.

Used for reproducibility and debugging.

---

# Registry Version

The version of the Style Registry.

Changes whenever registry behavior changes.

---

# Configuration Hash

A unique identifier representing the configuration used during analysis.

It allows exact reproduction of historical results.

---

# Browser-Only Architecture

An architecture where the complete analysis pipeline executes locally in the user's browser.

No backend services are required for the MVP.

---

# Deterministic

A property meaning that identical inputs always produce identical outputs.

Randomness is never used inside the Analysis Engine.

---

# Honest Result

A result that never invents information.

If a parameter cannot be determined confidently, it is reported as **Not detected** instead of guessed.

---

# Single Source of Truth

A design principle stating that every concept has exactly one authoritative definition.

Examples:

- Product behavior → PRD
- Styles → Style Registry
- Analysis pipeline → Analysis Engine
- Engineering decisions → Decisions Document

---

# Out of Scope

Functionality intentionally excluded from the MVP.

Examples include:

- MIDI export
- DAW integration
- Oscillator detection
- ADSR detection
- Source separation
- Plugin preset generation

These features may be added in future versions but are not considered implementation targets for v2.0.
